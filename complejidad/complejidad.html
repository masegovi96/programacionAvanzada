Notación Big O

¿Cómo medimos qué tan rápido (o lento) crece un algoritmo?

¿Qué es la complejidad de un algoritmo?

Cuando escribimos un algoritmo, nos interesa saber dos cosas:

Complejidad de Tiempo

¿Cuántas operaciones realiza el algoritmo conforme aumenta la cantidad de datos? No medimos segundos exactos (eso depende del hardware), sino cuánto crece el trabajo cuando hay más elementos.

Complejidad de Espacio

¿Cuánta memoria extra necesita el algoritmo? Algunos algoritmos son muy rápidos pero requieren mucha memoria adicional; otros hacen lo contrario.

La Notación Big O es la forma estándar de expresar esta complejidad. Describe el peor escenario posible (cota superior) del crecimiento del algoritmo en función del número de elementos n.

Por ejemplo: buscar un nombre en una lista

Imagina que tienes una lista de 1 000 nombres en un cuaderno y necesitas encontrar a "Zúñiga, Roberto".

O(1)

Alguien te da la página exacta donde está. Vas directo. No importa cuántos nombres haya, siempre tardas lo mismo.

Ej: acceder a un arreglo por índice: arr[42]

O(log n)

La lista está ordenada. Abres por la mitad: si el nombre va antes, descartas la mitad de atrás. Repites. Con 1 000 nombres bastan ~10 pasos.

Ej: búsqueda binaria, árboles AVL.

O(n)

La lista no está ordenada. Lees nombre por nombre desde el principio hasta encontrarlo. Con 1 000 nombres, en el peor caso revisas los 1 000.

Ej: búsqueda lineal, recorrer una lista.

O(n²)

Comparas cada nombre con todos los demás para ordenarlos. Con 1 000 nombres harías hasta 1 000 000 comparaciones.

Ej: Bubble Sort, Selection Sort.

Las notaciones y su significado

Para cada notación verás: qué significa, un ejemplo de código y cómo crece con distintos valores de n.

O(1)
Tiempo Constante

El algoritmo realiza siempre la misma cantidad de operaciones, sin importar cuántos datos haya. Es el ideal.

// Acceder a un elemento en un arreglo: siempre 1 operación
int elemento = arr[5];

// Insertar al inicio de una pila: siempre 1 operación
pila.push(x);

Estructuras: Pilas (push/pop), Colas (enqueue/dequeue), Tablas Hash (búsqueda promedio), acceso por índice en arreglos.

O(log n)
Tiempo Logarítmico

Con cada paso, el problema se reduce a la mitad. Es muy eficiente: duplicar los datos apenas agrega un paso extra.

// Búsqueda binaria en arreglo ordenado
int busquedaBinaria(int[] arr, int objetivo) {
    int izq = 0, der = arr.length - 1;
    while (izq <= der) {
        int mid = (izq + der) / 2;
        if (arr[mid] == objetivo) return mid;
        else if (arr[mid] < objetivo) izq = mid + 1;
        else der = mid - 1;
    }
    return -1;  // no encontrado
}

Estructuras: Árboles Binarios de Búsqueda (BST) balanceados, búsqueda binaria, Heap.

O(n)
Tiempo Lineal

El tiempo crece proporcionalmente con n. Si tienes el doble de datos, tarda el doble. Aceptable para la mayoría de casos.

// Recorrer una lista enlazada para encontrar un elemento
Nodo* buscar(Nodo* cabeza, int valor) {
    Nodo* actual = cabeza;
    while (actual != nullptr) {
        if (actual->dato == valor) return actual;
        actual = actual->siguiente;
    }
    return nullptr;
}

Estructuras: Listas Enlazadas (búsqueda), recorrido de árboles/grafos, Matrices (buscar un elemento).

O(n log n)
Tiempo Lineal-Logarítmico

Se aplica cuando hay que procesar cada uno de los n elementos, pero cada uno requiere una operación logarítmica (dividir y conquistar). Es la mejor complejidad posible para ordenar datos en general.

// Merge Sort — divide el arreglo a la mitad recursivamente
void mergeSort(int[] arr, int izq, int der) {
    if (izq < der) {
        int mid = (izq + der) / 2;
        mergeSort(arr, izq, mid);    // O(log n) niveles de recursión
        mergeSort(arr, mid+1, der);
        merge(arr, izq, mid, der);   // O(n) trabajo por nivel
    }
}

Algoritmos: Merge Sort, Quick Sort (promedio), Heap Sort, Shell Sort (mejor caso).

O(n²)
Tiempo Cuadrático

Generalmente resultado de dos ciclos anidados. Si duplicas los datos, el tiempo se cuadruplica. Funciona bien para listas pequeñas, pero se vuelve muy lento con datos grandes.

// Bubble Sort — compara cada par de elementos adyacentes
for (int i = 0; i < n; i++) {           // ciclo externo: n veces
    for (int j = 0; j < n - i - 1; j++) {  // ciclo interno: ~n veces
        if (arr[j] > arr[j+1]) {
            int temp = arr[j];
            arr[j] = arr[j+1];
            arr[j+1] = temp;
        }
    }
}

Algoritmos: Bubble Sort, Selection Sort, Insertion Sort (peor caso).

O(n+k) / O(nk)
Complejidad Especial (No Comparativa)

Estos algoritmos no comparan elementos entre sí, por lo que pueden superar la barrera teórica de O(n log n). Su complejidad depende tanto de n (cantidad de elementos) como de k (rango de valores o cantidad de dígitos). 

// Counting Sort — O(n + k)
// k = rango de valores (ej. 0-100)
int[] cuenta = new int[k + 1];
for (int x : arr) cuenta[x]++;   // O(n)
for (int i = 1; i <= k; i++)
    cuenta[i] += cuenta[i-1];     // O(k)
// Reconstruir arreglo: O(n)
// Radix Sort — O(n * k)
// k = número de dígitos del mayor número
// Por cada dígito aplica Counting Sort
for (int exp = 1; max/exp > 0; exp *= 10)
    countingSortPorDigito(arr, exp);
// Total: k pasadas × O(n) cada una

Condición: Counting Sort requiere que k (rango) sea razonablemente pequeño. Si k es muy grande, puede consumir más memoria que otros algoritmos.

¿Qué tomar en cuenta al elegir un algoritmo?

Tamaño de los datos

Para listas pequeñas (menos de 50 elementos), Insertion Sort puede ser más rápido en la práctica que Merge Sort a pesar de su peor complejidad, debido al bajo overhead.

¿Datos casi ordenados?

Insertion Sort tiene complejidad O(n) si los datos ya están casi ordenados. En ese escenario supera a Quick Sort y Merge Sort.

Tiempo vs. Espacio

Merge Sort es O(n log n) garantizado pero usa O(n) memoria extra. Heap Sort logra O(n log n) con O(1) espacio, sin necesitar memoria adicional.
