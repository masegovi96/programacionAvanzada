Peticiones HTTP

¿Qué es una petición HTTP?

HTTP (HyperText Transfer Protocol) es el protocolo de comunicación que define cómo los clientes y los servidores intercambian información en la web. Cada vez que tu navegador carga una página, obtiene una imagen o envía un formulario, está enviando una petición HTTP al servidor y esperando una respuesta HTTP.

Una petición HTTP es un mensaje estructurado que el cliente envía al servidor indicando qué recurso necesita y qué quiere hacer con él. El servidor procesa esa petición y devuelve una respuesta con el resultado.

Analogía: Una petición HTTP es como enviar una carta formal. Incluye la dirección del destinatario (URL), el tipo de solicitud (método), información adicional sobre quien pregunta (cabeceras) y, opcionalmente, el contenido del mensaje (cuerpo).

¿Cómo funciona?

Toda comunicación HTTP sigue el modelo petición–respuesta (request–response). El proceso tiene cuatro pasos principales:

El cliente inicia

El navegador o la aplicación construye una petición HTTP con método, URL, cabeceras y cuerpo (si aplica).

Viaja por la red

La petición viaja a través de internet hasta llegar al servidor indicado en la URL.

El servidor procesa

El servidor interpreta el método y la ruta, ejecuta la lógica correspondiente y prepara la respuesta.

El cliente recibe

El servidor devuelve una respuesta con un código de estado (ej. 200 OK) y los datos solicitados.

Anatomía de una petición HTTP

Toda petición está formada por tres partes:

POST /api/users HTTP/1.1
─────────────────────────── línea de solicitud (método + ruta + versión)

Host:         api.ejemplo.com
Content-Type: application/json
Authorization:Bearer eyJhbGciOiJIUzI1NiJ9...
─────────────────────────── cabeceras (headers)

{
  "name": "Ana López",
  "email": "ana@ejemplo.com"
}
─────────────────────────── cuerpo (body) — solo en POST, PUT, PATCH

Anatomía de una respuesta HTTP

HTTP/1.1 201 Created
─────────── línea de estado (versión + código + descripción)

Content-Type: application/json
Location:     /api/users/42
─────────── cabeceras de respuesta

{
  "id":    42,
  "name":  "Ana López",
  "email": "ana@ejemplo.com"
}
─────────── cuerpo de la respuesta

Códigos de estado más comunes

Código Nombre Significado
200OKLa petición fue exitosa y el servidor devuelve los datos.
201CreatedEl recurso fue creado correctamente.
204No ContentLa petición fue exitosa pero no hay datos que devolver (típico en DELETE).
400Bad RequestLa petición tiene un error de sintaxis o datos inválidos.
401UnauthorizedSe requiere autenticación para acceder al recurso.
403ForbiddenEl servidor entiende la petición pero no tiene permiso de ejecutarla.
404Not FoundEl recurso solicitado no existe en el servidor.
500Internal Server ErrorError genérico del servidor; algo salió mal en el backend.

Los métodos se clasifican en seguros (no modifican el servidor, ej. GET) e idempotentes (ejecutarlos varias veces produce el mismo resultado que ejecutarlos una sola vez, ej. PUT, DELETE).
GET

Obtener datos

GET solicita la lectura de un recurso. Es el método más utilizado: cada vez que escribes una URL en el navegador y presionas Enter, se envía un GET. No modifica nada en el servidor y no lleva cuerpo (body).

  • Es seguro e idempotente.
  • Los parámetros de filtrado van en la URL como query string: /posts?userId=1
  • Nunca debe usarse para enviar información sensible (los datos quedan en la URL y en los logs del servidor).
// Obtener todos los posts
GET https://jsonplaceholder.typicode.com/posts

// Obtener el post con id=1
GET https://jsonplaceholder.typicode.com/posts/1

// Filtrar posts del usuario 2
GET https://jsonplaceholder.typicode.com/posts?userId=2
POST

Crear un recurso

POST envía datos al servidor para crear un nuevo recurso. Los datos viajan en el body de la petición, generalmente en formato JSON. Cada llamada puede crear un recurso diferente, por lo que no es idempotente.

  • Es el método estándar para enviar formularios y subir información.
  • El servidor generalmente responde con 201 Created y devuelve el recurso recién creado.
  • No es seguro ni idempotente.
// Crear un nuevo post
POST https://jsonplaceholder.typicode.com/posts
Content-Type: application/json

{
  "title": "Mi nuevo post",
  "body":  "Contenido del post...",
  "userId": 1
}
PUT

Reemplazar un recurso

PUT reemplaza completamente un recurso existente con los datos que envías. Si omites un campo, ese campo se elimina o queda vacío en el servidor. Se usa cuando tienes la versión completa y actualizada del recurso.

  • Es idempotente: enviarlo dos veces produce el mismo resultado.
  • Requiere enviar todos los campos del recurso, no solo los que cambiaron.
  • El servidor generalmente responde con 200 OK y el recurso actualizado.
// Reemplazar completamente el post con id=1
PUT https://jsonplaceholder.typicode.com/posts/1
Content-Type: application/json

{
  "id":     1,
  "title": "Título completamente nuevo",
  "body":  "Cuerpo nuevo y completo",
  "userId":1
}
PATCH

Modificar parcialmente un recurso

PATCH aplica una modificación parcial a un recurso: solo envías los campos que quieres cambiar, el resto permanece igual. Es la diferencia clave con PUT.

  • Ideal para actualizar un campo sin necesidad de conocer todo el recurso.
  • Es más eficiente en red porque el body es más pequeño.
  • Generalmente devuelve 200 OK con el recurso actualizado.
// Cambiar solo el título del post con id=1
PATCH https://jsonplaceholder.typicode.com/posts/1
Content-Type: application/json

{
  "title": "Solo cambio el título"
}

// El body y el userId del post NO se modifican
DELETE

Eliminar un recurso

DELETE le indica al servidor que elimine el recurso identificado por la URL. Generalmente no lleva cuerpo en la petición.

  • Es idempotente: eliminar un recurso que ya no existe devuelve 404, pero el estado del servidor es el mismo.
  • El servidor responde típicamente con 204 No Content (sin body) o 200 OK con un mensaje de confirmación.
// Eliminar el post con id=1
DELETE https://jsonplaceholder.typicode.com/posts/1

// Respuesta del servidor:
HTTP/1.1 200 OK
{}  ← JSONPlaceholder devuelve un objeto vacío al eliminar
